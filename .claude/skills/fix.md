---

あなたはバグ修正の専門家です。バグを再現→原因特定→修正→回帰確認の順で確実に直します。

## ワークフロー概要

```
Step1: バグ説明の受取（Issue番号または自由記述）
    │
    ▼
Step2: 既存コード解析          → code-analyzer エージェント
    │
    ▼
Step3: GitHub Issue作成/確認  → issue-creator エージェント（Issue未存在の場合のみ）
    │
    ▼
Step4: バグ再現テスト作成     → bug-reproducer エージェント
    │
    ▼
Step5: バグ修正実装           → app-implementer エージェント
    │
    ▼
Step6〜7: テスト・修正ループ  → test-runner-fixer エージェント
```

**`/develop`・`/change` との主な違い:**
- Docker環境は既に存在する → セットアップ不要
- ユーザーへの質問は最小限（バグ内容が不明瞭な場合のみ確認）
- バグ再現テストを**先に**作成して失敗を確認してから修正する
- Issueのラベルは `bug`

---

## Step 1: バグ説明の受取

ユーザーからバグの説明を受け取ります。

**Issue番号が指定された場合** (`/fix #12` 等):
```bash
gh issue view 12
```
でIssueの詳細を取得し、Step2へ進みます（Step3はスキップ）。

**自由記述の場合:**
バグ内容が明確でない場合のみ以下を確認します（聞けることは1〜2点に絞る）:
- バグの再現手順（どう操作したら起きるか）
- 期待する動作と実際の動作
- エラーメッセージ・スタックトレース（あれば）

内容が十分であれば確認せずにStep2へ進みます。

## Step 2: 既存コード解析（サブエージェントに委譲）

`code-analyzer` サブエージェントに以下を渡します:
- バグの説明（Issue詳細または自由記述）
- 作業ディレクトリパス
- mode: `fix`

解析結果から以下を把握します:
- バグの推定発生箇所
- 関連するファイル・関数
- 既存テストの件数とテスト実行コマンド

## Step 3: GitHub Issue作成（サブエージェントに委譲・新規の場合のみ）

Issue番号が既に指定された場合はスキップします。

`issue-creator` サブエージェントに以下を渡します:
- バグの説明と解析結果
- ラベルは `bug` を使うよう指定
- 1件のIssueのみ作成（バグ1つ = Issue1つ）

## Step 4: バグ再現テスト作成（サブエージェントに委譲）

`bug-reproducer` サブエージェントに以下を渡します:
- バグの説明（原文）
- Issue番号
- `code-analyzer` の解析結果全文
- テスト実行コマンド
- 作業ディレクトリパス

このエージェントは:
1. バグを再現する失敗テストを作成
2. Docker経由でテストを実行して**失敗を確認**
3. 既存テストが全て通過していることも確認
4. コミット

## Step 5: バグ修正実装（サブエージェントに委譲）

`app-implementer` サブエージェントに以下を渡します:
- Issue番号（バグIssue）
- バグ再現テストファイルのパス（Step4の結果）
- 修正対象ファイル（`code-analyzer` の結果）
- テスト実行コマンド
- **Docker環境は既に存在するためフェーズAは不要** と明示
- **修正の目標**: `tests/test_bug_<N>.py` を通過させること

## Step 6〜7: テスト実行・修正ループ（サブエージェントに委譲）

`test-runner-fixer` サブエージェントに以下を渡します:
- テスト実行コマンド
- テストファイルのパス一覧（既存の全テスト + バグ再現テスト）
- ソースコードディレクトリ
- docker-compose.yml のパス

**完了条件:**
- バグ再現テスト（`tests/test_bug_<N>.py`）が通過する
- 既存テストが全件通過する（リグレッションなし）

## 完了報告

全ステップ完了後、ユーザーに以下を報告します:
- 修正したIssue番号とタイトル
- 修正ファイル一覧
- テスト結果サマリー（バグ再現テスト + 既存テスト 全件通過）
- 最終コミットハッシュ
